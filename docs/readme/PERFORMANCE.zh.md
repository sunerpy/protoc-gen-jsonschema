# 性能对比

> English version: [docs/PERFORMANCE.md](../PERFORMANCE.md)

本文对比为 Protobuf 消息获取 JSON Schema 的几种方式的运行时开销：运行时动态生成
与访问静态生成产物的差异。

## 总览

| 方案             | 延迟              | 吞吐量      | 内存分配  | 适用场景                     |
| ---------------- | ----------------- | ----------- | --------- | ---------------------------- |
| 动态生成         | ~8.4 μs           | ~118K ops/s | 2.7 KB/op | 大多数服务的默认选择         |
| 静态 `map` 常量  | ~0.29 ns + 5.9 μs | ~170K ops/s | 2.5 KB/op | 需要可变 map 结构            |
| 静态 Go 结构体   | ~0.29 ns + 4.7 μs | ~212K ops/s | 2.2 KB/op | 需要类型安全                 |
| 静态 JSON 字符串 | ~0.29 ns          | 无限制      | 0         | **最优** — HTTP 友好、零开销 |

静态访问比动态生成快约 **29,000 倍**，而 JSON 字符串常量本身就是 JSON，序列化开销为零。

## 静态格式对比

对于静态（`format=go_const`）输出，有四种可能的表示形式：

| 静态格式        | 访问      | 序列化   | 总耗时      | 适用场景         |
| --------------- | --------- | -------- | ----------- | ---------------- |
| JSON 字符串常量 | 0.29 ns   | 0 ns     | **0.29 ns** | HTTP API（最优） |
| Go 结构体常量   | 0.29 ns   | 4,708 ns | 4,708 ns    | 需要类型安全     |
| `map` 常量      | 0.29 ns   | 5,875 ns | 5,875 ns    | 需要可变结构     |
| 从文件读取      | 13,279 ns | 0 ns     | 13,279 ns   | 不推荐           |

## 规模化下的 CPU 开销

动态生成在不同请求速率下的每请求 CPU 开销：

| 场景     | RPS     | 动态方案 CPU   | 静态方案 CPU | 推荐         |
| -------- | ------- | -------------- | ------------ | ------------ |
| Web API  | 1,000   | 0.81%          | ~0%          | 动态         |
| 微服务   | 5,000   | 4.1%           | ~0%          | 动态         |
| API 网关 | 50,000  | 40.5%          | ~0%          | 静态         |
| 消息队列 | 500,000 | 405%（不可行） | ~0%          | 静态（必须） |

## 选型建议

- **动态生成** 适用于约 99% 的服务（RPS < 10K、需要运行时灵活性、作为库使用）。
  单次生成约 8 μs、约 2.7 KB。
- **静态生成** 适用于高吞吐路径（RPS > 100K、消息类型固定）。HTTP 响应优先使用
  JSON 字符串常量 —— 通过 `GetJSONSchemaBytes()` 零开销直接输出。
- 中等吞吐场景，可用动态生成 + 以消息全名为键的 `sync.Map` 缓存作为折中。

> 以上数字为单机基准测试的参考结果，应视为相对比例而非绝对保证。在你的硬件上运行
> `go test ./benchmark/...` 可获得实测数据。
